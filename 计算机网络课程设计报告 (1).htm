1 引言

• 学习和理解网络协议：直观观察协议交互过程

本课程设计旨在通过编程实现一个简单的网络嗅探器，将计算机网络的理论知识与编程实践相结合，加深对 TCP/IP 协议栈的理解。

2 相关技术与开发基础知识

2.1 原始套接字编程机制

套接字（Socket）是网络编程中应用层与传输层之间的抽象接口。标准的流式套接字（SOCK_STREAM）和数据报套接字（SOCK_DGRAM）只能处理传输层以上的数据，而原始套接字（Raw Socket, SOCK_RAW）允许程序直接发送和接收网络层的数据包，绕过系统协议栈的处理。

在 Windows 平台上，自 Windows XP SP2 以后对原始套接字的使用施加了严格限制：

• 原始套接字无法正常接收 TCP 数据包

• 原始套接字发送数据包时系统会自动填充 IP 头

• 创建原始套接字需要管理员权限

因此，在 Windows 上实现数据包捕获的常见方案是使用 WinPcap 或 Npcap 驱动。这些驱动在网卡驱动层插入一个中间层，将链路层的原始数据帧复制一份提供给用户态程序，绕过系统协议栈的干扰，从而实现真正的链路层数据包捕获。

2.2 TCP/IP 协议栈与报文格式

2.2.1 以太网帧格式

• 目的 MAC 地址（6 字节）：接收方的物理地址

• 源 MAC 地址（6 字节）：发送方的物理地址

• 以太网类型（2 字节）：标识上层协议类型，0x0800=IPv4, 0x0806=ARP, 0x86DD=IPv6

• 负载（46-1500 字节）：上层协议数据

• 帧校验序列 FCS（4 字节）：CRC 校验值

当以太网类型字段的值 ≤ 1500 时，该字段表示数据长度，此时帧格式为 IEEE 802.3。

2.2.2 IPv4 数据报格式

• 版本/首部长度（1 字节）：高4位=版本(4)，低4位=首部长度(×4字节)

• 总长度（2 字节）：整个 IP 数据报的长度

• 标识/标志/片偏移（4+2 字节）：用于 IP 分片重组

• 生存时间 TTL（1 字节）：每跳减1，为0时丢弃

• 协议（1 字节）：1=ICMP，6=TCP，17=UDP

• 源 IP 地址（4 字节）

• 目的 IP 地址（4 字节）

2.2.3 TCP 段格式

• 源端口（2 字节）、目的端口（2 字节）

• 序列号（4 字节）、确认号（4 字节）

• 数据偏移（4位）+ 保留位 + 标志位（6位：URG/ACK/PSH/RST/SYN/FIN）

• 窗口大小（2 字节）：流量控制

• 校验和（2 字节）：校验伪头部+TCP头部+数据

2.2.4 UDP 数据报格式

UDP 头部固定为 8 字节：源端口（2B）、目的端口（2B）、长度（2B）、校验和（2B）。

2.2.5 ICMP 报文格式

• 类型（1 字节）：0=Echo Reply, 8=Echo Request, 3=Destination Unreachable

• 代码（1 字节）：对类型的进一步细分

• 校验和（2 字节）

2.2.6 ARP 报文格式

• 硬件类型（2 字节）+ 协议类型（2 字节）

• 操作类型（2 字节）：1=请求，2=响应

• 发送方 MAC（6 字节）+ 发送方 IP（4 字节）

• 目标 MAC（6 字节）+ 目标 IP（4 字节）

2.3 字节序（Endianness）

网络协议标准规定所有多字节整型字段均使用大端序（Big-Endian）传输，而 x86 架构的 CPU 使用小端序（Little-Endian）。因此解析协议头部时必须使用 binary.BigEndian.Uint16() / Uint32() 方法进行字节序转换。

2.4 gopacket 库与 Npcap

gopacket 是 Google 开发的 Go 语言网络数据包处理库，提供 pcap 子包（封装 libpcap/Npcap C API）、layers 子包（预定义协议解码器）和 PacketSource（简化循环读取）。Npcap 是 Windows 平台的网络数据包捕获驱动，兼容 WinPcap API。

4.2 数据包捕获模块实现（Model层）

4.2.1 枚举网络接口

go
func ListInterfacesDetailed() ([]InterfaceInfo, error) {
    devices, err := pcap.FindAllDevs()
    if err != nil { return nil, fmt.Errorf("查找网络设备失败: %v", err) }
    var infos []InterfaceInfo
    for _, device := range devices {
        desc := device.Description
        if desc == "" { desc = device.Name }
        display := desc
        for _, addr := range device.Addresses {
            if addr.IP != nil {
                display += " [" + addr.IP.String() + "]"; break
            }
        }
        infos = append(infos, InterfaceInfo{Name: device.Name, Display: display})
    }
    if len(infos) == 0 { return nil, fmt.Errorf("未找到可用的网络接口") }
    return infos, nil
}

4.2.2 创建捕获器

NewCapturer 通过 pcap.OpenLive 在混杂模式下打开网卡：

go
func NewCapturer(ifaceName string) (*Capturer, error) {
    handle, err := pcap.OpenLive(ifaceName, 65536, true, 30*time.Second)
    if err != nil { return nil, fmt.Errorf("打开接口失败: %v (需管理员权限)", err) }
    handle.SetBPFFilter("")  // 捕获所有包
    source := gopacket.NewPacketSource(handle, handle.LinkType())
    source.DecodeOptions = gopacket.Lazy  // 延迟解码
    return &Capturer{handle: handle, source: source, iface: ifaceName}, nil
}

4.3 协议分析模块实现（Model层）

4.3.1 顶层入口：Analyze 方法

Analyze 方法解析以太网帧头部的14字节，然后根据以太网类型分发：

go
func (a *Analyzer) Analyze(data []byte) AnalyzeResult {
    result := AnalyzeResult{Length: len(data)}
    if len(data) < 14 { result.Detail = "数据包长度不足"; return result }
    result.DstMAC = net.HardwareAddr(data[0:6]).String()
    result.SrcMAC = net.HardwareAddr(data[6:12]).String()
    etherType := binary.BigEndian.Uint16(data[12:14])
    if etherType <= 1500 {
        result.EthernetType = "IEEE 802.3"
    } else { switch etherType {
        case 0x0800: result.EthernetType = "IPv4"
            a.parseIPv4(data[14:], &result)
        case 0x0806: result.EthernetType = "ARP"
            a.parseARP(data[14:], &result)
        case 0x86DD: result.EthernetType = "IPv6"; result.Protocol = "IPv6"
        case 0x8100: result.EthernetType = "VLAN"; result.Protocol = "VLAN"
    }}
    return result
}

4.3.2 IPv4 数据报解析

go
func (a *Analyzer) parseIPv4(data []byte, result *AnalyzeResult) {
    if len(data) < 20 { return }
    ihl := int(data[0]&0x0F) * 4
    result.SrcIP = net.IP(data[12:16]).String()
    result.DstIP = net.IP(data[16:20]).String()
    result.Length = int(binary.BigEndian.Uint16(data[2:4]))
    switch data[9] {
    case 1:  result.Protocol = "ICMP"; a.parseICMP(data[ihl:], result)
    case 6:  result.Protocol = "TCP";  a.parseTCP(data[ihl:], result)
    case 17: result.Protocol = "UDP";  a.parseUDP(data[ihl:], result)
    default: result.Protocol = fmt.Sprintf("IP-%d", data[9])
    }
}

4.3.3 TCP 段解析

go
func (a *Analyzer) parseTCP(data []byte, result *AnalyzeResult) {
    if len(data) < 20 { return }
    srcPort := binary.BigEndian.Uint16(data[0:2])
    dstPort := binary.BigEndian.Uint16(data[2:4])
    result.SrcPort, result.DstPort = fmt.Sprintf("%d", srcPort), fmt.Sprintf("%d", dstPort)
    flags := data[13]; flagStr := ""
    if flags&0x02 != 0 { flagStr += "SYN " }
    if flags&0x10 != 0 { flagStr += "ACK " }
    if flags&0x01 != 0 { flagStr += "FIN " }
    if flags&0x04 != 0 { flagStr += "RST " }
    if flags&0x08 != 0 { flagStr += "PSH " }
    detail := fmt.Sprintf("Seq=%d Win=%d", binary.BigEndian.Uint32(data[4:8]),
        binary.BigEndian.Uint16(data[14:16]))
    if flagStr != "" { detail += fmt.Sprintf(" [%s]", flagStr) }
    if app := identifyAppProtocol(srcPort, dstPort); app != "" { detail += " " + app }
    result.Detail += " | " + detail
}

func (a *Analyzer) parseUDP(data []byte, result *AnalyzeResult) {
    if len(data) < 8 { return }

    if app := identifyAppProtocol(srcPort, dstPort); app != "" { result.Detail += " | " + app }
}

4.3.6 ARP 报文解析

go
func (a *Analyzer) parseARP(data []byte, result *AnalyzeResult) {
    if len(data) < 28 { return }
    result.SrcIP, result.DstIP = net.IP(data[14:18]).String(), net.IP(data[24:28]).String()
    result.Protocol = "ARP"
    op := "请求"
    if binary.BigEndian.Uint16(data[6:8]) == 2 { op = "响应" }
    result.Detail += fmt.Sprintf(" | ARP%s %s->%s", op, result.SrcIP, result.DstIP)
}

6 开发过程中遇到的问题及解决办法

6.1 Windows 平台原始套接字限制

问题：Windows 对原始套接字实施了安全限制，无法直接使用 SOCK_RAW 接收 TCP 数据包。解决：采用 gopacket/pcap 库，底层调用 Npcap 驱动绕过系统协议栈，直接从网卡驱动层获取链路层帧。

6.2 接口名映射问题

问题：pcap.FindAllDevs() 返回的设备名是 GUID 格式，直接传描述名给 OpenLive 会失败。解决：定义 InterfaceInfo 结构体同时保存 Name（设备名）和 Display（显示名），用户选择显示名，底层使用设备名。

6.3 字节序解析错误

问题：直接读取 []byte 中的 uint16 得到错误数值（如 80 变成 20480）。解决：网络字节序为大端序，使用 binary.BigEndian.Uint16() / Uint32() 正确转换。

6.4 数据包越界 panic

问题：数据包长度不足时访问切片下标导致运行时 panic。解决：在每一层解析函数入口检查最小长度，长度不足时直接 return。

6.5 管理员权限

问题：普通用户运行程序时 pcap.OpenLive 返回权限错误。解决：以管理员身份运行，错误信息中明确提示“需要管理员权限”。

7 待解决的问题及改进方向

7.1 现有系统待解决的问题

• IPv6 支持不完善：仅能识别 IPv6 数据包，未详细解析 IPv6 头部和扩展头部

• IP 分片重组：未实现分片重组，仅能解析单个分片

• 应用层深度解析：仅通过端口号识别，未解析 HTTP/DNS/TLS 等协议数据

• IEEE 802.3 帧格式解析不完整

7.2 系统优化与改进方向

• GUI 优化：使用前端框架增加饼图、折线图等可视化元素

• BPF 过滤：支持用户输入过滤表达式，提高分析效率

• PCAP 导出：支持导出为标准 pcap 格式文件

• 流量统计分析：按协议分布、TOP N 通信对等维度分析

• 异常流量告警：支持自定义规则的实时告警

• IPv6 深度解析：实现 IPv6 头部和 ICMPv6 解析
